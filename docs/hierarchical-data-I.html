Google provides a free public GA4 dataset in BigQuery derived from their own online store (merch.google.com). The data covers real shopping behaviour — page views, product interactions, and purchases — but user identifiers have been obfuscated for privacy.

1 The Challenge: Why Is GA4 Data Hierarchical?

1.1 From Flat Files to Event Streams

Traditional web analytics tools exported flat CSV files: one row per session, one column per metric. GA4 takes a fundamentally different approach. It records every user interaction as an event, and because different event types carry completely different attributes, a flat structure is impractical.

Consider the variety:

  • A page_view event carries: page URL, page title, referrer, scroll depth
  • A purchase event carries: transaction ID, revenue, coupon codes, items purchased
  • A click event carries: link URL, link text, element ID

If these lived in a flat table, you would need hundreds of nullable columns. BigQuery’s nested and repeated field types solve this elegantly — but require us to “rectangle” the data before we can analyse it.

1.2 GA4’s Three Nesting Patterns

GA4 data uses three distinct patterns of nesting. Understanding which pattern applies to each column is the key to choosing the right rectangling tool:

GA4 Nesting Patterns and Their Rectangling Tools
Column(s) BigQuery Type R List-Column Shape Rectangling Tool
device, geo, traffic_source, app_info RECORD (struct) List of named lists (1 per row) unnest_wider()
items RECORD REPEATED (array of named structs) List of tibbles with named columns unnest_longer() + unnest_wider()
event_params, user_properties RECORD REPEATED (array of key-value structs) List of tibbles with key + typed value columns unnest_longer() + coalesce() + pivot_wider()1
1 Pattern 3 is not covered in R4DS Ch. 23 and is the main supplementary topic of this document.

4 Part 1: Applying R4DS Rectangling to GA4

NoteR4DS Chapter 23 Coverage

The techniques in this section are directly taught in R4DS Chapter 23. If you have worked through that chapter, this section applies those tools to real GA4 data. If not, this is your introduction to the rectangling toolkit.

Key functions: unnest_wider(), unnest_longer(), hoist() (all from {tidyr})

4.1 What Is Rectangling?

Rectangling is the process of transforming a deeply nested hierarchical data structure (like a list-column) into a flat, rectangular data frame where every column contains atomic values and every row represents one observation.

The core insight from R4DS is that there are really only two fundamental operations:

Operation When to use tidyr function
“Widen” a struct Each list element is a named list (one record with named fields) unnest_wider()
“Lengthen” an array Each list element is a vector/list of multiple things (an array) unnest_longer()

Most complex cases combine both operations in sequence.

4.2 Pattern 1: Simple Structs with unnest_wider()

GA4’s device, geo, and traffic_source columns are all non-repeated RECORD types — each row contains exactly one record with a fixed set of named fields. This is the simplest nesting pattern.

Tipnames_sep for Avoiding Name Collisions

When unnesting multiple struct columns that might share field names, use names_sep to prefix each new column:

```{r}

ga4 |>
  unnest_wider(device, names_sep = "_") |>   # → device_category, device_browser
  unnest_wider(geo,    names_sep = "_")       # → geo_country, geo_city
```

4.3 Pattern 2: Arrays of Named Structs with unnest_longer() + unnest_wider()

The items column contains e-commerce line items. Unlike device, this is a repeated RECORD — a purchase can have multiple items. Each row’s items cell contains a tibble with one row per item.

This is the pattern covered in R4DS Chapter 23 using gh_repos as the example: an array where each element is a named struct.

4.3.1 Ch23

  • When the list is unnamed, use unnest_longer()
  • When the list is name, use unnest_wider()

4.3.2 Understanding the Two-Step Process

Code 
# Step 0: examine the items column for a purchase event
purchase_rows <- which(ga4$event_name == "purchase")
ga4 |>
  filter(row_number() == purchase_rows[1]) |>
  select(event_name, items) |>
  unnest(items) # Quick look

unnest() unnest automatically by using a combination of both unnest_wider and unnest_longer. It is not good for learning.

Compare the code above with the one below.

Code 
ga4 |>
  filter(row_number() == purchase_rows[1]) |>

5 Part 2: Beyond R4DS — The Key-Value Pattern

ImportantGA4-Specific Content — Not in R4DS Ch. 23

This section covers a pattern that R4DS Chapter 23 does not demonstrate: the key-value struct array. This is the most complex and most important nesting pattern in GA4 data, used by event_params and user_properties.

Understanding this section is the central skill for GA4 analytics in R.

5.1 The Anatomy of event_params

In R4DS, the nested data examples (like got_chars or gh_repos) have named fields at each level. When you unnest a gh_repo, you get columns like id, name, and stargazers_count because those are the field names in the struct.

GA4’s event_params is fundamentally different. Instead of named fields, it stores parameters in a generic key-value format:

```{r}
# What we WISH event_params looked like (named struct — covered by R4DS):
event_params = { page_location = "...", session_id = 123, engagement_time = 456 }

# What event_params ACTUALLY looks like (key-value struct — NOT in R4DS):
event_params = [
  { key = "page_location",   value = { string_value = "https://...", int_value = NULL, ... } },
  { key = "ga_session_id",   value = { string_value = NULL, int_value = 123,            ... } },
  { key = "engagement_time", value = { string_value = NULL, int_value = 456,            ... } }
]
```

Why did Google design it this way? Because different event types send completely different parameters. Using a generic key-value store means the schema never changes — new parameters can be added to any event without altering the table structure.

The second complication: the value field is itself a struct with four sub-fields — one for each data type that a parameter value might be:

```{r}
value.string_value  ← populated if the value is text
value.int_value     ← populated if the value is an integer
value.float_value   ← populated if the value is a decimal
value.double_value  ← populated if the value is a high-precision decimal
```

For any given parameter, exactly one of these will be non-NULL. We must consolidate them into a single value column.

  filter(!map_lgl(user_properties, \(x) is.null(x) || nrow(x) == 0)) |>
  distinct(user_pseudo_id, .keep_all = TRUE) |> # One record per user
  unnest_longer(user_properties) |>
  unnest_wider(user_properties) |>
  unnest_wider(value, names_sep = "_") |>
  unnest_wider(value_1) |>
  mutate(
    prop_value = coalesce(
      as.character(string_value),
      as.character(int_value),
      as.character(float_value),
      as.character(double_value)
    )
  ) |>
  select(user_pseudo_id, event_date, key, prop_value) |>
  pivot_wider(
    names_from = key,
    values_from = prop_value,
    values_fn = first
  )

# The Challenge: Why Is GA4 Data Hierarchical?

## From Flat Files to Event Streams

Traditional web analytics tools exported flat CSV files: one row per session, one column per metric. GA4 takes a fundamentally different approach. It records every user interaction as an **event**, and because different event types carry completely different attributes, a flat structure is impractical.

Consider the variety:

-   A `page_view` event carries: page URL, page title, referrer, scroll depth
-   A `purchase` event carries: transaction ID, revenue, coupon codes, items purchased
-   A `click` event carries: link URL, link text, element ID

If these lived in a flat table, you would need hundreds of nullable columns. BigQuery's **nested and repeated field types** solve this elegantly — but require us to "rectangle" the data before we can analyse it.

## GA4's Three Nesting Patterns

GA4 data uses three distinct patterns of nesting. Understanding which pattern applies to each column is the key to choosing the right rectangling tool:

# Part 1: Applying R4DS Rectangling to GA4 {#part1}

::: callout-note
## R4DS Chapter 23 Coverage

The techniques in this section are **directly taught in R4DS Chapter 23**. If you have worked through that chapter, this section applies those tools to real GA4 data. If not, this is your introduction to the rectangling toolkit.

**Key functions**: `unnest_wider()`, `unnest_longer()`, `hoist()` (all from `{tidyr}`)
:::

## What Is Rectangling?

**Rectangling** is the process of transforming a deeply nested hierarchical data structure (like a list-column) into a flat, rectangular data frame where every column contains atomic values and every row represents one observation.

The core insight from R4DS is that there are really only two fundamental operations:

| Operation | When to use | tidyr function |
|--------------------|------------------------|----------------------------|
| "Widen" a struct | Each list element is a **named list** (one record with named fields) | `unnest_wider()` |
| "Lengthen" an array | Each list element is a **vector/list of multiple things** (an array) | `unnest_longer()` |

Most complex cases combine both operations in sequence.

## Pattern 1: Simple Structs with `unnest_wider()` {#pattern1}

GA4's `device`, `geo`, and `traffic_source` columns are all **non-repeated RECORD types** — each row contains exactly one record with a fixed set of named fields. This is the simplest nesting pattern.

### The `device` Column

```{r}
#| label: unnest-device

device_data <- ga4 |>
  select(user_pseudo_id, event_name, event_date, device) |>
  unnest_wider(device) # ← expands named list into columns

glimpse(device_data) # 13 columns unnested
```

::: callout-note
The `device` list-column is replaced by individual columns (13 of them): `category`, `mobile_brand_name`, `browser`, `operating_system`, etc. Each column name matches the original struct field name.
:::

::: callout-tip
## `names_sep` for Avoiding Name Collisions

> When unnesting multiple struct columns that might share field names, use `names_sep` to prefix each new column:

```{r}
# Prevents collisions if two structs both have, e.g., a "version" field
ga4 |>
  unnest_wider(device, names_sep = "_") |>   # → device_category, device_browser
  unnest_wider(geo,    names_sep = "_")       # → geo_country, geo_city
```
:::

### The `geo` Column

## Pattern 2: Arrays of Named Structs with `unnest_longer()` + `unnest_wider()` {#pattern2}

The `items` column contains **e-commerce line items**. Unlike `device`, this is a *repeated* RECORD — a purchase can have multiple items. Each row's `items` cell contains a tibble with one row per item.

This is the pattern covered in R4DS Chapter 23 using `gh_repos` as the example: an array where each element is a named struct.

### Ch23

-   When the list is unnamed, use `unnest_longer()`
-   When the list is name, use `unnest_wider()`

### Understanding the Two-Step Process

```{r}
#| label: items-structure

# Step 0: examine the items column for a purchase event
purchase_rows <- which(ga4$event_name == "purchase")
ga4 |>
  filter(row_number() == purchase_rows[1]) |>
  select(event_name, items) |>
  unnest(items) # Quick look
```

::: callout-note
`unnest()` unnest automatically by using a combination of both `unnest_wider` and `unnest_longer`. It is not good for learning.

Compare the code above with the one below.
:::

```{r}
ga4 |>
  filter(row_number() == purchase_rows[1]) |>
  select(event_name, items) |> # unnamed list
  unnest_longer(items) |> # tibble: 1 × 2 named list
  unnest_wider(items) # tibble: 1 × 27
```

> The pipeline for an array of named structs:

```{r}         
items (list-column of tibbles)
  └─ unnest_longer(items)  → one row per item, items column now a single tibble-row
       └─ unnest_wider(items)  → each item field becomes its own column
```

```{r}
#| label: unnest-items

items_flat <- ga4 |>
  filter(event_name == "purchase") |>
  select(event_date, user_pseudo_id, event_timestamp, items) |>
  unnest_longer(items) |> # Step 1: one row per item
  unnest_wider(items) # Step 2: expand item struct fields into columns

glimpse(items_flat)
```

::: callout-note
## Why Two Steps?

`unnest_longer()` turns the *array* into individual rows (each element becomes its own row). After this step, the `items` column contains a single struct — which we then "widen" with `unnest_wider()`.

You can often shortcut with just `unnest(col)`, which does both steps for the common case of a list-column of data frames. But being explicit teaches the structure.
:::


## Pattern 1b: `hoist()` for Selective Extraction {#hoist}

> When you only need **a few fields** from a large struct, `hoist()` is more efficient than a full unnest because it avoids creating columns you don't need.

::: callout-note
## R4DS Coverage

> `hoist()` is covered in R4DS Section 23.4. The syntax differs slightly from `unnest_wider()`
>
> You name the output columns and specify the path to each value.
:::

```{r}
#| label: hoist-example

# Extract only country and device category without unnesting everything
ga4_targeted <- ga4 |>
  select(event_name, event_date, user_pseudo_id, device, geo) |>
  #unnest_wider(device)
  hoist(device, device_category = "category", browser = "browser") |>
  #unnest_wider(geo)
  hoist(geo, country = "country", city = "city")

glimpse(ga4_targeted)
```

> `hoist()` leaves the original list-columns in place (now with the extracted fields removed). This is useful when you want to keep the nested column for later processing while also having quick access to the most important fields.

# Part 2: Beyond R4DS — The Key-Value Pattern {#part2}

::: callout-important
## GA4-Specific Content — Not in R4DS Ch. 23

This section covers a **pattern that R4DS Chapter 23 does not demonstrate**: the *key-value struct array*. This is the most complex and most important nesting pattern in GA4 data, used by `event_params` and `user_properties`.

Understanding this section is the central skill for GA4 analytics in R.
:::

## The Anatomy of `event_params`

> In R4DS, the nested data examples (like `got_chars` or `gh_repos`) have **named fields** at each level. When you unnest a `gh_repo`, you get columns like `id`, `name`, and `stargazers_count` because those are the field names in the struct.
>
> GA4's `event_params` is fundamentally different. Instead of named fields, it stores parameters in a **generic key-value format**:

```{r}         
# What we WISH event_params looked like (named struct — covered by R4DS):
event_params = { page_location = "...", session_id = 123, engagement_time = 456 }

# What event_params ACTUALLY looks like (key-value struct — NOT in R4DS):
event_params = [
  { key = "page_location",   value = { string_value = "https://...", int_value = NULL, ... } },
  { key = "ga_session_id",   value = { string_value = NULL, int_value = 123,            ... } },
  { key = "engagement_time", value = { string_value = NULL, int_value = 456,            ... } }
]
```

**Why did Google design it this way?** Because different event types send completely different parameters. Using a generic `key-value` store means the schema never changes — new parameters can be added to any event without altering the table structure.

**The second complication**: the `value` field is itself a struct with four sub-fields — one for each data type that a parameter value might be:

```{r}         
value.string_value  ← populated if the value is text
value.int_value     ← populated if the value is an integer
value.float_value   ← populated if the value is a decimal
value.double_value  ← populated if the value is a high-precision decimal
```

> For any given parameter, exactly one of these will be non-NULL. We must *consolidate* them into a single value column.

### Stage 1: `unnest_longer()` — Expand the Array

> The first stage turns the array into rows. Each event's parameter list becomes individual rows, one per parameter:

```{r}
#| label: stage-1-unnest-longer

stage1 <- toy_params |>
  unnest_longer(event_params)

cat("After Stage 1:", nrow(stage1), "rows x", ncol(stage1), "cols\n")
stage1 |> select(event_id, event_name, event_params) #  15 rows x 3 cols
```

> After this step, `event_params` is no longer a list of tibbles — it is now a single tibble-row per event parameter. But the `value` inside each row is still a nested named list.

### Stage 1b: `unnest_wider()` — Expand the Inner Struct

> Now flatten the key and value struct fields into columns:

```{r}
#| label: stage-1b-unnest-wider

stage1b <- stage1 |>
  unnest_wider(event_params) |> # expands key, value into columns
  unnest_wider(value, names_sep = "_") |> # expands value struct sub-fields
  unnest_wider(value_1)
cat("After Stage 1b:", nrow(stage1b), "rows x", ncol(stage1b), "cols\n") #15 rows x 7 cols
stage1b
```

> We now have `key`, `string_value`, `int_value`, `float_value`, `double_value` as separate columns. But each row has only **one non-NULL value** across the four typed columns.

### Stage 2: `coalesce()` — Consolidate the Value Types

::: callout-important
## This Step Is Not in R4DS

`coalesce()` returns the first non-NULL value across its arguments. Here we use it to merge the four typed value columns into a single `value` column. This is a GA4-specific requirement not demonstrated in R4DS Chapter 23.
:::

```{r}
#| label: stage-2-coalesce

stage2 <- stage1b |>
  mutate(
    value = coalesce(
      string_value,
      as.character(int_value),
      as.character(float_value),
      as.character(double_value)
    )
  ) |>
  select(event_id, event_name, key, value) # Keep only what we need

cat("After Stage 2:", nrow(stage2), "rows x", ncol(stage2), "cols\n")
stage2
```

> Now we have a clean long-format table: one row per parameter per event, with a single `value` column.

### Stage 3: `pivot_wider()` — Reshape to Analytical Format

> The final step converts from long format (one row per parameter) to wide format (one column per parameter), giving us the flat, analysis-ready structure:

```{r}
#| label: stage-3-pivot-wider

stage3 <- stage2 |>
  pivot_wider(
    names_from = key,
    values_from = value,
    values_fn = first # If a key appears more than once, take the first
  )

cat("After Stage 3:", nrow(stage3), "rows x", ncol(stage3), "cols\n") #2 rows x 8 cols
stage3
```

> We now have one row per event and one column per parameter — exactly the flat structure we need for analysis.

## Pattern 3 Summary

The **key-value struct pattern** requires three stages that build on each other:

```         
List-column of tibbles          (event_params before)
   │
   ▼  unnest_longer()           → Long format: one row per parameter
   │
   ▼  unnest_wider() ×2         → Columns for key + value sub-fields
   │
   ▼  coalesce()                → Single unified value column
   │
   ▼  pivot_wider()             → Wide format: one column per parameter
   │
Flat rectangular tibble         (event_params after)
```

This pipeline applies identically to `event_params` and `user_properties`.

| Task | Code |
|----------------------|-----------------------------------------------|
| Start session | `con <- dbConnect(duckdb())` then `CREATE VIEW ga4 AS ...` |
